# General Agent Workspace Template

A folder-based template for human–AI collaborative workspaces. Designed for use with Claude Code (or any agentic CLI), it defines a consistent layout so slash commands, subagents, and MCP tools can reference stable relative paths.

> **Note:** `.template-meta/` is about the template itself — ignore it when working on a project built from this template.

## How to use this template

This repo is marked as a GitHub **template repository**, so you can spin up a new workspace from it in two ways.

### Option A — Cloud first (GitHub UI)

1. On this repo's GitHub page, click the green **"Use this template"** → **"Create a new repository"**.
2. Pick an owner, name, and visibility:
   - **Public** — leave visibility as Public.
   - **Private** — tick Private.
3. Create the repo, then clone it locally:
   ```bash
   git clone git@github.com:<owner>/<new-repo>.git
   cd <new-repo>
   ```

### Option B — Local first (`gh` CLI)

Creates the GitHub repo from the template, clones it, and sets `origin` in one shot.

**Public:**
```bash
gh repo create <new-repo> \
  --template danielrosehill/General-Agent-Workspace-Template \
  --public --clone
cd <new-repo>
```

**Private:**
```bash
gh repo create <new-repo> \
  --template danielrosehill/General-Agent-Workspace-Template \
  --private --clone
cd <new-repo>
```

### First commit

The template clone already contains an initial commit from the template. To mark your starting point:

```bash
git commit --allow-empty -m "chore: initialize workspace from template"
git push -u origin main
```

Then run `/setup` inside Claude Code to bootstrap the project brief.

### Option C — Slash command (`/new-from-template`)

This repo ships a Claude Code slash command that wraps Option B: it prompts for name/visibility/destination, calls `gh repo create --template`, clones locally, and makes an initial commit.

**Install it into your user-scope commands** (so it's available in any Claude Code session, not just inside this repo):

```bash
cp .claude/commands/new-from-template.md ~/.claude/commands/
```

Then from any Claude Code session, run:

```
/new-from-template
```

and answer the prompts. The command lives at [`.claude/commands/new-from-template.md`](.claude/commands/new-from-template.md).

## Philosophy

- **Folder-based** — state lives on disk, not in chat history
- **Separate workspaces, shared contract** — humans and agents have distinct folders, but the point of the separation is *streamlined collaboration*, not walls. The contract is what makes handoffs cheap.
- **Open-ended on MCPs** — optional integrations (Pinecone, Gemini transcription, GitHub Gist, etc.) are used if present, otherwise the local filesystem is the source of truth
- **Slash commands orchestrate** — commands in `.claude/commands/` reference the paths below

## How it operates

Two sides, one contract:

- **Human writes to** `inputs/` (prompts, voice notes, raw context drops) and `context/for-agent/` (long-form SOPs and instructions).
- **Agent writes to** `outputs/`, `management/`, `docs/for-user/`, `docs/for-world/`, and curates `context/store/`.
- **`STATUS.md`** at the repo root is the live snapshot both sides read first. It's auto-generated by `/status:refresh` from the `management/` and `inputs/` folders — never hand-edited.

A typical session:

1. **Start** — agent reads `STATUS.md`: active plans, open blockers, pending prompts, last log. No folder crawling.
2. **Work** — human drops prompts/voice notes in `inputs/`; agent executes them, writing artifacts to `outputs/runs/<slug>/`, logs to `management/agent-logs/`, and plans to `management/plans/active/` for non-trivial work.
3. **Refresh** — `/status:refresh` regenerates `STATUS.md` from the current folder state.
4. **End** — if the next session is a *different agent or editor* (or a fresh session after a crash/context-full), run `/handover:handoff`. Otherwise `STATUS.md` alone carries state forward. Handovers are strictly for bridging discontinuities — not for general status or session summaries.

**Context vs docs** — these look similar, they're not. `context/` is the RAG-style background the agent pulls on when reasoning (chunked, retrievable). `docs/` is discrete documents meant to be read top-to-bottom by a named audience.

## Folder contract

The canonical folder contract lives in [`CLAUDE.md`](CLAUDE.md) — it is the single source of truth for which folder does what and who writes to it. The summary below is for orientation only; when in doubt, read `CLAUDE.md`.

```
inputs/
  prompts/
    pending/       # drop prompt/task files here for batch execution
    edits/         # reusable edit-round prompts for iterative revisions
    completed/     # processed prompts are moved here
  audio/           # raw voice notes to be transcribed

context/
  to-add/          # raw context the user drops in for ingestion
  store/           # chunked, organized background context (RAG-style)
  api-docs/        # API reference material for the agent to consult
  for-agent/       # long-form SOPs / instructions the user writes for the agent

outputs/
  runs/            # initial run artifacts (date- or subject-based subfolders)
  combined/        # aggregated / concatenated repeat runs
  pdf/             # Typst-rendered PDFs of combined runs

management/
  project-brief/   # overarching project brief, goal, stack choices
  plans/           # discrete UUID-named work plans, sorted by state:
    draft/           # being written, not yet approved
    active/          # approved and in progress
    completed/       # finished
    archived/        # abandoned or superseded
  blockers/        # documented blockers (UUID-named)
  debug-recipes/   # reusable debug recipes (UUID-named, tagged)
  sessions/        # session summaries (timestamp-named)
  feature-requests/# feature requests (UUID-named)
  stack/           # current stack documentation (current.md)
  agent-logs/      # progress logs written by the agent
  handovers/       # archived handover docs between sessions/agents
  decisions/       # decision records (ADR-style)

docs/
  for-user/        # docs the agent writes to explain things to the user
  for-world/       # public-facing drafts (optionally published via Gist MCP)

code/
  <project-name>/  # source code MUST be nested at least one level deep
```

## Code nesting rule

For code projects, source code always lives under `code/<project-name>/` — never loose at the repo root, never loose directly in `code/`. This keeps the organisational framework separate from the codebase and lets a single workspace wrap multiple sub-projects. See `code/example-project/` as a placeholder.

## Context storage

The local `context/store/` folder is the default persistent context.

**If a Pinecone MCP is available**, the agent uses it as the *primary* context store and mirrors key entries to `context/store/` for offline reference. Without Pinecone, `context/store/` is authoritative and the agent reads/writes it directly — no other behavior changes. See `CLAUDE.md` § Context retrieval for the exact contract.

## Slash commands

Commands live under `.claude/commands/` and are grouped by namespace. Invoke them as `/<namespace>:<command>`.

**Workspace**
- `/setup` — first-time workspace bootstrap via interview subagent or voice note
- `/init-project` — capture the initial project brief into `management/project-brief/`
- `/organise-repo` — audit and enforce the template's folder conventions
- `/archive-handover` — copy root `HANDOVER.md` into `management/handovers/` before `/handover:resume` consumes it

**Status** (`status:*`)
- `/status:refresh` — regenerate the root `STATUS.md` snapshot

**Plans** (`plan:*`)
- `/plan:new` — create a UUID-named plan in `management/plans/draft/`
- `/plan:update` — transition a plan between state folders

**Runs** (`run:*`)
- `/run:new` — start a new output run (date- or subject-based)
- `/run:prompts` — execute all pending prompts in `inputs/prompts/pending/`
- `/run:prompt` — execute a single targeted prompt (from `pending/` or `edits/`)
- `/run:combine` — aggregate repeat runs into `outputs/combined/`
- `/run:to-pdf` — render combined runs to PDF via Typst

**Context** (`ctx:*`)
- `/ctx:chunk` — ingest raw files from `context/to-add/` into `context/store/`
- `/ctx:process-audio` — transcribe files in `inputs/audio/`, extract context + prompts

**Documentation** (`doc:*` — all read from `templates/` and fill them in)
- `/doc:blocker` — capture a blocker into `management/blockers/`
- `/doc:decision` — capture a decision into `management/decisions/`
- `/doc:stack` — write/update `management/stack/current.md`
- `/doc:debug` — capture a reusable debug recipe into `management/debug-recipes/`
- `/doc:session` — capture a session summary into `management/sessions/`
- `/doc:feature` — capture a feature request into `management/feature-requests/`
- `/doc:edit-round` — turn edit instructions into a plan in `management/plans/draft/`

**Logging** (`log:*`)
- `/log:progress` — append a progress log to `management/agent-logs/`

### Handover plugin

Session handover uses the [Claude-Handover plugin](https://github.com/danielrosehill/Claude-Handover) — install once:
```
/plugin install handover@danielrosehill/Claude-Handover --scope user
```
Then use `/handover:create`, `/handover:resume`, or `/handover:handoff`. Run `/archive-handover` between create and resume to preserve a copy in `management/handovers/`.
